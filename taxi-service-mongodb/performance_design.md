# Анализ производительности и стратегии оптимизации

## 1. Анализ производительности

### 1.1 Hot paths (часто используемые операции)

| Операция | Endpoint | Частота | Обоснование |
|----------|----------|---------|-------------|
| Получение активных заказов | GET /api/v1/rides/active | Очень высокая | Водители постоянно обновляют список, чтобы найти новый заказ |
| Поиск пользователя по логину | GET /api/v1/users/{login} | Высокая | Каждый запрос аутентификации и профиля |
| Создание заказа | POST /api/v1/rides | Высокая | Основное действие пассажиров |
| Принятие заказа | POST /api/v1/rides/{ride_id}/accept | Высокая | Водители соревнуются за заказы |
| История поездок | GET /api/v1/users/{user_id}/rides | Средняя | Пользователи проверяют свои поездки |

### 1.2 Медленные операции

| Операция | Причина | Ожидаемая задержка |
|----------|---------|---------------------|
| Поиск по маске имени | Полное сканирование, regex операции | 50-200 мс |
| Получение истории поездок | JOIN с rides, сортировка | 30-100 мс |
| Создание заказа | Вставка + индексация | 10-30 мс |
| Получение активных заказов | Фильтрация по статусу | 20-50 мс |

### 1.3 Требования к производительности

| Метрика | Значение | Обоснование |
|---------|----------|-------------|
| Время отклика (p95) | < 100 мс для кешируемых, < 500 мс для остальных | UX стандарт |
| Пропускная способность | 1000 RPS для чтения, 100 RPS для записи | Пиковая нагрузка такси |
| Доступность | 99.9% | Критический сервис |

## 2. Стратегия кеширования

### 2.1 Какие данные кешировать

| Данные | Причина | Приоритет |
|--------|---------|-----------|
| Пользователь по логину | Частые чтения, редкие изменения | Высокий |
| Результаты поиска по маске | Тяжелый запрос | Высокий |
| Активные заказы | Очень частые чтения | Высокий |
| История поездок | Частые чтения конкретного пользователя | Средний |

### 2.2 Выбор стратегии

**Cache-Aside (Lazy Loading)**

Применяется для GET запросов:
1. Проверяем кеш
2. Если есть — возвращаем
3. Если нет — берем из БД, сохраняем в кеш

Почему: Простота реализации, контроль над заполнением кеша, нет лишних данных.

**Write-Through для инвалидации**

При обновлении данных (POST, PUT):
1. Обновляем БД
2. Удаляем связанные записи из кеша

### 2.3 TTL (Time To Live)

| Тип данных | TTL | Обоснование |
|------------|-----|-------------|
| Пользователь по логину | 300 секунд | Данные меняются редко (смена имени — нечасто) |
| Результаты поиска | 300 секунд | Не критично к устареванию на 5 минут |
| Активные заказы | 10 секунд | Должны быть свежими для водителей |

### 2.4 Ключи кеша

| Endpoint | Формат ключа | Пример |
|----------|--------------|--------|
| GET /users/{login} | `user:login:{login}` | `user:login:john_doe` |
| GET /users/search | `user:search:{first_name}:{last_name}` | `user:search:John:Doe` |
| GET /rides/active | `rides:active` | `rides:active` |

### 2.5 Инвалидация кеша

| Действие | Инвалидируемые ключи |
|----------|----------------------|
| Создание заказа | `rides:active` |
| Принятие заказа | `rides:active` |
| Завершение заказа | `rides:active` |
| Обновление пользователя | `user:login:*`, `user:search:*` |

## 3. Стратегия Rate Limiting

### 3.1 Какие endpoints лимитировать

| Endpoint | Причина |
|----------|---------|
| GET /api/v1/users/search | Самый тяжелый запрос, защита от брутфорса |
| GET /api/v1/rides/active | Предотвращение избыточных запросов от водителей |

### 3.2 Выбор алгоритма

**Token Bucket**

Почему:
- Простая реализация
- Позволяет кратковременные всплески (burst)
- Не требует хранения большого количества данных
- Хорошо работает для API с разной нагрузкой

### 3.3 Лимиты

| Endpoint | Лимит | Обоснование |
|----------|-------|-------------|
| GET /api/v1/users/search | 10 запросов в минуту | Защита от брутфорса, тяжелый запрос |
| GET /api/v1/rides/active | 10 запросов в минуту | Достаточно для обновления раз в 6 секунд |

### 3.4 HTTP ответ при превышении

```json
{
    "error": "Too many requests. Rate limit exceeded.",
    "limit": 10,
    "remaining": 0,
    "reset_seconds": 57
}
```

Статус: **HTTP 429 Too Many Requests**

## 4. Реализация

### 4.1 Кеширование (Redis)

```cpp
// Проверка кеша
std::string cache_key = cache::UserSearchKey(first_name, last_name);
auto& redis = cache::RedisClient::GetInstance();
auto cached_data = redis.Get(cache_key);

if (cached_data.has_value()) {
    return cached_data.value();  // Cache HIT
}

// Cache MISS — запрос к MongoDB
auto result = db_query();

// Сохранение в кеш
redis.Set(cache_key, json_response, 300);
```

### 4.2 Rate Limiting (Token Bucket)

```cpp
class TokenBucket {
    bool TryConsume(int tokens) {
        Refill();  // Добавляем токены за прошедшее время
        if (tokens_ >= tokens) {
            tokens_ -= tokens;
            return true;
        }
        return false;
    }
};
```

## 5. Ожидаемый эффект

### 5.1 Кеширование

- Снижение нагрузки на MongoDB на 60-80% для чтения
- Уменьшение времени ответа с 50-200 мс до 1-5 мс для кешируемых запросов
- Более стабильное время ответа при пиковых нагрузках

### 5.2 Rate limiting

- Защита от случайных DDoS и ошибок в клиентах
- Справедливое распределение ресурсов
- Предотвращение утечек данных через массовые запросы
